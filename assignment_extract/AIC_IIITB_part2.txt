--- PAGE 23 ---
The Evaluation Metric — RMSE on log(SalePrice)
Kaggle scores your submission with RMSE between the log of the predicted price and the log of the observed price:
Why the log?
Relative error matters more than absolute error
A $20k error on a $100k house → 20% off → bad
A $20k error on a $700k house → 3% off → fine
Taking the log makes the metric penalise percentage errors equally across all price ranges
The practical trick
Predict log(SalePrice) directly → then a standard RMSE on log-prices is exactly the Kaggle score.
Easier to optimise — and the leaderboard score is identical.

--- PAGE 27 ---
Skewness — and Why We Care
What is skewness?
A number that measures the asymmetry of a distribution:
Skew = 0 → symmetric, bell-shaped (like a Gaussian)
Skew > 0 (right-skewed) → long tail on the right — a few large values pull the mean up
Skew < 0 (left-skewed) → long tail on the left — a few small values pull the mean down
Why we want a near-Gaussian target
1. Linear models (Ridge, Lasso) assume residuals ≈ Gaussian — easier to satisfy when y itself is symmetric
2. Stabilises variance: errors become relative (% of the price), not absolute — same fairness across cheap and luxury
houses
3. Compresses outliers: extreme luxury prices no longer dominate the loss
4. Matches the Kaggle metric: log-RMSE is already a log score → training on log(y) directly optimises the right
thing
Tree models (RF, XGBoost, ...) are less sensitive to target shape — but the log transform still helps them via point 4
(metric alignment).

Block 4 : Why Regularise? — The Multicollinearity Problem
Recall from EDA
We found 3 pairs of strongly correlated features:
GarageCars  GarageArea (0.88)
GrLivArea  TotRmsAbvGrd (0.83)
TotalBsmtSF  1stFlrSF (0.82)
The problem for OLS
When two features carry the same information,  becomes nearly singular → its inverse explodes → coefficients become unstable and can flip
sign on tiny data changes.
Example — same model re-trained on slightly different subsets of the same data:
Subset 1 Subset 2
β_GarageCars +12 000 −3 000
β_GarageArea −25 +40
Wildly different coefficients — not trustworthy.
The fix — regularisation
Add a penalty on large coefficients to keep them small and stable:
→ This is the idea behind Ridge (L2 penalty) and Lasso (L1 penalty).

Ridge Regression — L2 Penalty
The objective
Plain OLS minimises:
Ridge adds a squared penalty on the coefficients:
What this does
Shrinks all coefficients toward 0 — the large ones especially
But coefficients are never exactly 0 (the squared term is smooth)
Result: stable, bounded coefficients, even with multicollinearity
The hyperparameter α (alpha)
α Behaviour
0 No penalty → identical to plain OLS
small (e.g. 0.1 ) Light shrinkage
large (e.g. 100 ) Heavy shrinkage → coefficients near 0 → underfit
→ ∞ All coefficients pushed to 0
→ The optimal α is data-dependent. We tune it with GridSearchCV in Block 5.

--- PAGE 41 ---
Lasso Regression — L1 Penalty
The objective
Lasso adds an absolute-value penalty on the coefficients:
Compare with Ridge: penalty was  (squared, not absolute value).
The key difference vs Ridge
Like Ridge: shrinks coefficients toward 0
Unlike Ridge: pushes useless coefficients exactly to 0 → automatic feature selection
Why? The V-shape (see next slide)
The L1 penalty has a constant gradient near 0 (because of the absolute value's sharp corner at 0) → strong enough to push small coefficients all the
way to 0. Ridge's smooth parabola loses its push near 0, so coefficients stay slightly non-zero.
→ Result: a sparse model — only the most useful features survive.
Lasso = Least Absolute Shrinkage and Selection Operator — the "selection" is literally in the name.

Ridge vs Lasso — The Geometry of the Penalty
# Visualize the two penalty shapes (markdown)
beta = np.linspace(-2, 2, 200)
ridge_penalty = beta ** 2
lasso_penalty = np.abs(beta)
fig, ax = plt.subplots(figsize=(8, 5))
ax.plot(beta, ridge_penalty, label="Ridge (L2): β²", lw=2.5)
ax.plot(beta, lasso_penalty, label="Lasso (L1): |β|", lw=2.5)
ax.axvline(0, color='gray', lw=0.5)

What you see
Ridge (L2) = smooth parabola → the curve flattens near 0 → the "push" toward 0 weakens
Lasso (L1) = sharp V → the slope is constant (±1) → the push toward 0 stays strong
→ That's why Lasso pushes coefficients exactly to 0, but Ridge only near 0.

--- PAGE 44 ---
ElasticNet — The Best of Both Worlds
When you cannot decide between L1 and L2, use both with a tunable mix.
The combined penalty
Two hyperparameters
Parameter What it controls Range
alpha Overall strength of the penalty [0, ∞)  — higher = stronger shrinkage
l1_ratio Mix between L1 and L2 [0, 1]  — 0 = pure Ridge, 1 = pure Lasso, 0.5 = balanced
When to use it
You suspect some features can be safely dropped (the L1 part handles this — sets them to 0)
And the remaining features may be correlated with each other (the L2 part handles this — distributes weight evenly across correlated features)
Best of both worlds: feature selection + graceful handling of multicollinearity
Slightly slower to tune (2 hyperparameters instead of 1)
sklearn usage
from sklearn.linear_model import ElasticNet
ElasticNet(alpha=0.001, l1_ratio=0.5, max_iter=10000)
 A great candidate for your group's personal extension — run a GridSearch over both alpha and l1_ratio, and compare the result with
your Ridge and Lasso scores.

How Are These Models Actually Trained?
OLS and Ridge — closed-form (matrix solution)
For plain OLS and for Ridge, there is an exact mathematical formula that gives the optimal coefficients in a single matrix calculation:
→ sklearn.linear_model.LinearRegression computes this directly. One shot, no iteration.
Lasso, SVM, MLP — iterative methods
For these models, no closed-form exists. sklearn uses an iterative algorithm:
1. Start with random coefficients
2. Compute how wrong the predictions are (the loss)
3. Compute the gradients of the loss with respect to each coefficient
4. Slightly adjust the coefficients in the direction that decreases the loss
5. Repeat until the loss stops improving
→ This idea — gradient descent — is the workhorse of modern ML, from Lasso to ChatGPT.
Methods you will meet:
Lasso → coordinate descent (updates one coefficient at a time)
MLP → gradients computed via backpropagation (chain rule), weights updated by an optimizer (Adam, SGD…)

Bias-Variance Tradeoff — The Concept
The total prediction error of a model decomposes into three terms:
Bias² — systematic error from a model that is too simple to capture the real relationship (underfitting)
Variance — error from a model that sticks too closely to the specific training set, and gives wildly different predictions if retrained on slightly different
data (overfitting)
Irreducible noise — the random part of the data that no model can predict
Term Meaning High when...
Bias Systematic error from oversimplification Model is too simple → underfitting
Variance Sensitivity to training data fluctuations Model is too complex → overfitting
Irreducible noise Random noise in the data itself Always present — cannot reduce
The tradeoff
More complex model → less bias, but more variance
Less complex model (more regularisation) → more bias, but less variance
→ Regularisation deliberately adds bias to reduce variance.
→ Net effect on total error is positive when the right balance is found.
Irreducible noise:
Imagine two identical houses in the dataset — same size, same neighbourhood, same year. They can still sell at different prices because of
negotiation, timing, the seller's urgency, or the buyer's taste. None of our 79 features captures these random factors. So even a perfect model would
still make this error — it is the absolute floor of our prediction error."

from sklearn.pipeline      import Pipeline
from sklearn.impute        import SimpleImputer
from sklearn.preprocessing import StandardScaler, OneHotEncoder

# Numerical: impute median + scale
num_pipe = Pipeline([
    ("imputer", SimpleImputer(strategy="median")),
    ("scaler",  StandardScaler()),
])

# Combine: apply each sub-pipeline to the right columns
preprocessor = ColumnTransformer([
    ("num", num_pipe, num_cols),
    ("cat", cat_pipe, cat_cols),
])

Numerical features

Categorical features

one-hot encode
All fit operations happen inside the CV
folds → no data leakage

Why Cross-Validation?
The problem with a single train/test split
Split 1458 houses 80/20 once → one single score, unreliable:
Different random seed → different score
And we "waste" 20% of our training data
K-Fold CV — the robust alternative
Split into 5 equal folds, then for each fold: train on the other 4, predict on this one.
[ Val  | Train | Train | Train | Train ]  → score 1
[ Train| Val   | Train | Train | Train ]  → score 2
[ Train| Train | Val   | Train | Train ]  → score 3
[ Train| Train | Train | Val   | Train ]  → score 4
[ Train| Train | Train | Train | Val   ]  → score 5
                                            ───────
                                            mean ± std
→ Robust score: mean ± std. The standard deviation tells us how much we can trust the mean.
Why not just submit each model to Kaggle?
Kaggle limits submissions per day (10)
One submission = one number on a fixed test set → easy to overfit "the leaderboard"
CV is fast, free, unlimited, and provides an uncertainty estimate
 
CV is our internal scorecard. We compare models on CV → submit only the winner to Kaggle.

Ridge + GridSearchCV — How It Works
Ridge — quick reminder (from Block 4)
Linear regression + L2 penalty on the coefficients:
One hyperparameter: alpha controls the strength of the penalty
High alpha → strong shrinkage → simpler model → left of the U-curve → high bias (underfit)
Low alpha → close to OLS → complex model → right of the U-curve → high variance (overfit)
Goal: find the α at the bottom of the U-curve (best bias-variance trade-off)
GridSearchCV — automated hyperparameter search
Instead of trying alpha values one by one by hand:
1. Define a grid of candidate values, e.g. [0.01, 0.1, 1, 10, 30, 100]
2. For each value → run K-fold CV → get a CV score (the mean of the K fold scores)
3. Pick the value with the best CV score → that's our best_alpha
4. (Bonus) GridSearchCV automatically refits the model on the full training set with the winning α
→ Same recipe for any hyperparameter, on any model.

Decision Tree — How It Works
A decision tree is a flowchart of yes/no questions that progressively narrows down the prediction.
A toy tree on House Prices
                GrLivArea > 1500?
                  /         \
                Yes          No
                /             \
       OverallQual > 7?    avg = $145k
         /        \
       Yes         No
        /           \
   avg = $290k   avg = $190k
Each leaf stores the average SalePrice of all training houses that fell into it.
Prediction
For a new house: follow the questions top-down → reach a leaf → return its average.
→ Example: GrLivArea = 2000, OverallQual = 8 → right, then right → predicted $290k.
How the tree picks its splits
At each node, the algorithm tries every feature and every threshold, and picks the split that most reduces the variance of the target
inside the two child nodes (creates the most homogeneous groups, each child group contains prices more similar to each other than
in the parent group).
A single tree is fast and interpretable, but easily overfits if grown too deep. → Solution: combine many trees.

--- PAGE 76 ---
Random Forest — Bagging Many Decorrelated Trees
A single deep tree overfits. The fix: train many trees and average their predictions.
Bagging — "Bootstrap Aggregating"
Two sources of randomness make each tree slightly different:
1. Random rows (bootstrap sample drawn with replacement) → each tree sees a different subset of houses
2. Random features at each split → no tree can always rely on the same dominant feature (e.g., OverallQual)
Prediction
Each tree votes → final prediction = average of all tree predictions.
   new house  →  tree 1   predicts $245k
                 tree 2   predicts $260k      → average = $251k
                 tree 3   predicts $248k         ↑
                 ...                              final
                 tree 100 predicts $251k         prediction
Why it works
Individual trees are high-variance (each overfits its bootstrap sample)
But the trees overfit in different directions → averaging cancels out the random errors
Same low bias as individual trees, but much lower variance → better generalisation
Same intuitive trees + a clever combination = no overfit. The first form of ensemble learning.

Why Averaging Reduces Variance — A Closer Look
Which variance? — The bias-variance decomposition
For a fixed input , the total prediction error at that point decomposes into three terms:
The variance we want to reduce = dispersion of individual predictions around their own mean, not around the true value.
Concrete example — house actually sold at $210k
Tree Prediction
1 $215k
2 $180k
3 $230k
... ...
100 $205k
Average 
Note: variance is computed around  (the mean prediction), not around the true value $210k. The $10k gap between mean and truth = the bias.
→ The standard deviation (√variance) is ≈ $30k around the mean — that's how much individual tree predictions deviate from each other.
→ Each individual tree's prediction sits at roughly ±$30k around the mean (not around the truth!).
What averaging does
→ If we re-ran 100 different forests, their averaged predictions would only vary by ±$3k — 10× more stable than any individual tree.
The forest's bias is the same as a single tree. Its variance is 100× smaller. That's the whole point of Random Forest. 77

How They Differ — A Closer Look
 Tree growth strategies
Level-wise (XGBoost) — fills the tree one full level at a time, all leaves at the same depth. Balanced, slower to converge, less overfit.
Leaf-wise (LightGBM) — at each step, splits the leaf with the highest loss reduction, regardless of its depth. Faster convergence, but trees
become asymmetric and overfit more easily.
Symmetric / Oblivious (CatBoost) — at each level, all nodes use the same split feature and threshold. Very restrictive structure → built-in
regularisation + extremely fast prediction.
 Categorical handling — what each really does
XGBoost: does NOT one-hot encode for you. You must encode categoricals manually (one-hot or label or target encoding) before passing the
data.
LightGBM: has a categorical_feature parameter — handles encoding internally, but less robust than CatBoost.
CatBoost: pass categorical columns as-is via cat_features=[...]. No manual encoding needed. Internally uses ordered target encoding.
 Ordered boosting (CatBoost specific)
Standard target encoding leaks information about the target into the features. CatBoost prevents this: at training step , the target encoding for
sample  is computed only from samples that come before  in a random ordering — never including 's own target. → No data leakage during
training.

--- PAGE 80 ---
Target Encoding & CatBoost's Solution to Data Leakage
 What is target encoding?
Replace each category by the mean of the target for rows in that category — turns text into a meaningful number, no one-hot explosion.
Row Neighborhood SalePrice
1 OldTown $140k
2 OldTown $160k
3 OldTown $155k
4 OldTown $130k
→ mean("OldTown")  = (140 + 160 + 155 + 130) / 4 = $146.25k → every "OldTown" row becomes 146.25
 The data leakage problem
When encoding row 3, row 3's own target ($155k) is included in the mean. The feature now "knows" the target → the model cheats during training → overfits badly on test.
 CatBoost's solution — Ordered Target Encoding
Random permutation of rows. For each row, the encoding uses only the rows that come before it:
Row Rows used to compute encoding Encoding
1 (none — first in order) global prior
2 row 1 140
3 rows 1, 2 (140 + 160) / 2 = 150
4 rows 1, 2, 3 (140 + 160 + 155) / 3 = 151.7
→ Row 3's own target ($155) is never used to encode row 3. No leakage by construction.
CatBoost uses a different random permutation for each boosting round → no single ordering biases the model. All this happens invisibly when you set cat_features .

What TargetEncoder does (anti-leakage built in)
Each category → a single numeric value = mean of the target across rows with that category
Internal 5-fold cross-fitting (cv=5 ) → for each fold, encoding is computed from the OTHER 4 folds → no leakage during training

smooth="auto"  → Empirical Bayes shrinkage: rare categories are pulled toward the global mean (acts as the "prior")
